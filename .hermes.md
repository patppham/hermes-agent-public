# Public Hermes Runtime Guidance

This file is a safe template for a private Hermes deployment. It describes
orchestration behavior only. Keep real paths, account routing, credentials,
source manifests, memory records, and household policy in a private runtime.

## Execution

Choose one owner for a task. Batch independent discovery, make dependent edits,
run the narrowest meaningful verification, and stop when the success criteria
are met. Delegate only when work is clearly broad, sustained, or independently
reviewable. Keep final synthesis, secrets, authentication, destructive actions,
and ambiguous judgment with the orchestrator.

Use deterministic code for projection, reconciliation, health checks, and
watchdogs. Use a judgment-capable agent only after code has bounded and verified
the context. Provider and model choices belong in private configuration, not in
public prompt contracts.

## Context and memory

Prefer current, source-backed context over remembered claims. Every adapter
should preserve freshness, missing-source, cancellation, and confidence
metadata. Do not treat an empty result as proof that a source has no data until
the source-specific empty-result check has run.

Memory is an input with a privacy scope, not a universal scratchpad. Load only
the memory and skills required for the task. Never copy raw messages, documents,
credentials, or private indexes into public artifacts or broad model context.

## State and mutations

Canonical state owns commitments and completion. Generated projections are
derived views and must not be hand-edited. Reconciliation must be deterministic,
idempotent, and auditable. It may repair a canonical status when the evidence
is unambiguous, but it must not become an idea generator or silently create a
new task.

External actions require an explicit approval gate, an allowlisted adapter, and
a destination chosen by private configuration. A public reference runtime may
authorize a destination in a result, but must not execute the action.

## Routing and delivery

Keep private-owner work on a private lane and shared-household work on a shared
lane only when the output has been deliberately redacted for that audience.
No-op when there is no material change, no useful question, or no authorized
recipient. Separate job execution failures, source freshness failures, and
delivery failures so a green process check cannot hide a missed message.

## Skills and verification

Load the smallest relevant skill set. Skills define contracts and safety
boundaries; adapters provide credentials and real integrations. Keep cron
prompts, skills, and the context builder aligned when cadence, routing,
approval, or mutation behavior changes.

Before publishing a private mirror, run the public-surface validator, unit
tests, compilation checks, JSON/YAML parsing, and a secret/PII scan. Keep
generated state, logs, sessions, caches, auth files, local databases, and
vendored dependencies out of the public repository.
